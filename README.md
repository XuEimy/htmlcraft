# 🎨 HTMLCraft

**✨ 一套让 AI 在生成前端时遵守设计规范的指令系统。**

---

## 🔭 基于什么现象

> **Vibe Coding**：用自然语言让 AI 生成前端代码的开发方式。

AI 能写出能跑的前端代码，但产出的页面在设计层面高度不稳定。具体表现为五个可观察、可复现的现象：

**🌐 跨模型漂移**
同一句需求给不同模型，出来的页面完全不一样。Claude 做的和 GPT 做的风格对不上。

**💨 上下文蒸发**
同一个模型在长对话中，前面定好的设计方向会被后面的内容冲掉。第 5 轮还好，第 20 轮就开始跑偏。

**💥 Agent 碰撞**
多个 AI agent 各做一个 section，拼在一起不像同一个页面。Hero 是极简风，Feature 变成了 SaaS 模板，Footer 又是另一种风格。

**🪄 迭代意外修改**
用户说"把标题改大一点"，模型改了标题，顺手把间距、配色、按钮样式也改了——用户没要求这些改动。

**🎲 模糊需求随机产出**
用户说"高端一点""科技感"，每次出来的结果都不一样，因为这些词没有确定性的设计映射。

---

## 🔍 根因是什么

AI 能执行设计（写 CSS、排布局），但在执行之前没有经过设计决策。它没有一个独立于对话上下文的、跨模型通用的设计规则文件来约束它。

**💡 代码能力够了，设计约束缺失了。**

---

## 🎯 解决什么问题

不是"让 AI 生成更好的代码"——代码能力已经够了。

而是在代码生成之前插入一层设计决策系统，把设计规则从模型的"理解"中抽离出来，变成一套独立的、可被任何执行者读取的文件。然后让这套文件在整个项目生命周期中充当锚点——

不管换模型、换工具、换 session、换 agent，规则始终在那里。

---

## 🧩 为什么是 Skill

**Skill 的本质：一份在 AI 开始工作之前就被注入上下文的规则文件。**

它解决的是 AI 前端生成的三个根本问题：

**🧠 不让 AI 自己"猜"设计**
"高端""科技感""现代感"这种词，AI 可以有无数种解读，每次结果都不一样。Skill 把这些模糊词翻译成具体的 token 值——色值、字号、字重、间距、圆角——AI 照着执行，没有歧义。

**🔗 跨模型、跨工具、跨 session 保持一致**
换 Claude 还是 GPT，用 Cursor 还是 Codex，开新 session 还是接着聊——规则文件是同一份，AI 一进来就读，读完就知道该怎么设计，而不是各做各的。

**🛡️ 给 AI 设一道"写代码之前"的门槛**
不是等 HTML 生成完了再让 AI 自检，而是先锁定视觉论点、确认设计系统、生成规格文件，再进入真正的构建阶段。设计决策在前，代码生成在后。

其他方案为什么不行：

- 🖥️ 做一个独立网站给用户找灵感？用户不会在 vibe coding 流程中跳出去开另一个页面
- 📄 写一份设计文档发给团队？人可以读，但 AI 不会主动读一份放在别处的文档
- 🎨 做一个 Figma 插件？Figma 不在 vibe coding 的工作流里
- 🏗️ 做一个 CSS 框架（类似 Tailwind）？框架约束的是实现层，不是决策层。它能保证按钮都是 8px 圆角，但不能保证页面该用什么节奏、什么字体关系、什么视觉重力

## 🔀 完整使用流程

使用流程分成两条线：**设计工作流**（AI 怎么用 HTMLCraft 做项目）和**本地工具流**（怎么把 dashboard 跑起来看状态）。

---

### 第一条线：设计工作流

#### 从零开始的新项目

**第一步，触发 HTMLCraft。**
用 `/htmlcraft` 或 `/using htmlcraft` 开始。它先不会直接写代码，而是先判断你属于哪种情况。

**第二步，Preflight 扫描。**
检查当前项目目录里有没有这 6 个核心交付物：

```
.htmlcraft.md
design-spec.md
design-tokens.css
design-tokens.json
html-handoff.md
index.html
```

全部没有 → **System Build Mode**（全新项目）
有一部分 → **Partial Audit Mode**
全部齐了 → **Audit / Revision Mode**（不再从头问需求）

**第三步，确认输出类型。**

| 类型 | 核心侧重 |
|---|---|
| 🌐 网页 | Hero + CTA |
| 📝 HTML 报告 | 信息层级 + 打印阅读 |
| 📽️ HTML PPT | 全屏节奏 + 翻页感 |

**第四步，需求摄入。**
一轮收齐：主题 / 目标用户 / 核心目标 / 必须有的内容 / 明确不要什么 / 品牌 Logo 状态 / 设备（手机/电脑/都要）/ 本地还是部署 / 有无视觉参考 / 反信号（明确不喜欢的风格）。

**第五步，输出 PRD 确认。**
把需求整理成结构化摘要：一句话定位、目标用户、核心场景、唯一任务、必须有、明确不做、成功标准。你确认后才进入下一阶段。

**第六步，参考依据确认。**
三种情况：

- **有截图 / URL** → 先拆解：布局、排版层级、艺术感来源、留白节奏、色彩关系、材质感、动效预期。再做三轮确认：主要参考哪一张、分析对不对、到底想参考什么。
- **完全没参考** → 明确告知：视觉方向只能从 PRD 推导，风险是更容易模板化。

**第七步，锁定视觉论点。**
给出 2~3 个完全不同的方向候选，每个候选讲清楚：

```
这个页面住在什么样的世界里
为什么适合你的 PRD
Hero 用什么原型
字体气质是什么
色彩哲学是什么
艺术处理怎么做
会对 token 产生什么影响
风险是什么，怎么避免掉进 AI 通用模板
```

最后必须收敛成 **1 个主论点**，不是保留多个方向。

**第八步，锁定 Hero 原型。**
网页项目进一步明确：archetype 是哪种 / 第一屏要让用户感受到什么 / 视觉锚点在哪 / 文案是宣言型、问题型还是沉浸型 / CTA 是强推动还是轻引导 / 第一屏和第二屏怎么衔接。

**第九步，设计系统确认。**

Group A：**排版 + 色彩**
→ display font 和 body font 的关系 / 字号层级 / 混排策略 / 冷暖色调 / accent 用法

Group B：**组件 + 动效**
→ 按钮高度 / 卡片用 border 还是 shadow / 导航固定还是静态 / 图标 family / 整体动效风格（quiet / crisp / cinematic）

确认前必须过一次**设计律令核对**——检查字体数量、颜色槽数量、按钮高度、图标规则、标题层级、正文行长是否违反硬规则。

**第十步，生成规格文件。**
这是核心产物，最重要是这 5 个：

| 文件 | 作用 |
|---|---|
| `design-tokens.css` | **第一真值**——颜色/字体/间距/圆角从 token 来，HTML 不写死 |
| `design-tokens.json` | 机器可读镜像，供工具链消费 |
| `design-spec.md` | 完整设计规格，人读版本 |
| `html-handoff.md` | 约束 HTML 映射方式 |
| `.htmlcraft.md` | 项目记忆文件，新会话/新模型接手先读它 |

**第十一步，确认是否进入构建。**
提醒你把规格文件和 index.html 放在同一目录，以后"按规格修改"。确认后才进入真正写页面的阶段。

**第十二步，HTML 映射。**
按顺序构建：

```
引入 design-tokens.css → 搭布局 shell → 做 Hero → 做 section 主体 →
做 CTA / Footer / 组件 → 补交互状态和响应式 → 最后 polish
```

**硬规则：HTML 里不允许硬编码颜色、间距、圆角、字体值，必须引用 CSS 变量。**

**第十三步，修改影响评估。**
每次你说"帮我改一下"，AI 必须先输出影响评估：

```
改什么 / 当前值是什么 / 目标值是什么 / 影响到哪些区块 /
是不是触及设计系统 / 会否破坏视觉论点 /
到底是只改 token，还是 token + HTML 一起改
```

**第十四步，审计回执。**
每轮修改完成后，报告：

```
本轮修改是什么 / 通过了没 / 新增通过项 / 新增风险项 / 未解决项 /
Hero 有没有偏移 / Design Tokens 有没有漂移 /
组件一致性和响应式有没有出问题
```

连续修改 3 轮以上 → 要求补跑一次完整审计。

**第十五步，最终审计。**
没有审计通过，就不算完成。按顺序读取文件，然后检查：

```
视觉论点还在不在 / 设计规范和页面结构匹不匹配 /
HTML 有没有绕开 token 写死值 /
Hero、CTA、按钮、卡片、图标是否一致 /
有没有重新滑回 AI 通用模板感
```

---

#### 已有项目：Audit-Only Mode

如果目录里已经有 `.htmlcraft.md`、`design-tokens.css`、`design-spec.md` 等交付物，不会再从 PRD 和参考图开始。直接读取这些文件 → 出审计结果。

如果要改，就走：**审计 → 修改影响评估 → 修改 → 审计回执 → 必要时完整复核**。

---

### 第二条线：本地工具流（Dashboard）

**第一步，** 进入 `htmlcraft/` 项目根目录。

**第二步，** 运行 `start-dashboard.sh`。
脚本自动检查 Node、npm、python3 是否存在，检查端口 3000 有没有被占用，必要时自动安装 dashboard 依赖。

**第三步，** 启动后打开 `http://localhost:3000`。

**第四步，** dashboard 监听根目录关键文件变化，自动更新页面状态，必要时重新触发 Python 审计。

dashboard 是用来**看项目现在进行到哪一步、交付物齐不齐、有没有漂移**的，不是用来做设计的。

---

**一句话总结整个流程：**

```
🆕 新项目：需求 → PRD → 参考图 → 视觉论点 → 设计系统 → 规格文件 → HTML → 审计
📦 老项目：读交付物 → 审计 → 修改 → 再审计
```

## 📦 核心交付物

每个项目生成 5 个文件：

| 📄 文件 | 🎯 作用 |
|---|---|
| `design-tokens.css` | 第一真值。所有颜色、间距、字体变量。HTML 只能引用变量，不能硬编码 |
| `design-tokens.json` | 机器可读镜像，供工具链消费 |
| `design-spec.md` | 完整设计规格，人读版本 |
| `html-handoff.md` | 实现交接文档，约束 HTML 映射方式 |
| `.htmlcraft.md` | 项目上下文入口。新会话、新模型读这一个文件就能接手，不用重走设计流程 |

---

## 🖥️ 支持的输出类型

- **🌐 网页**：landing page、产品页、作品集
- **📝 HTML 报告**：调研报告、白皮书、分析文档（图文并茂，支持打印）
- **📽️ HTML PPT**：演示幻灯片（全屏翻页、键盘/触摸控制）

## 🚀 使用方式

在支持 OpenClaw 的环境中，直接调用：

```
/htmlcraft
```

HTMLCraft 会引导你完成：PRD 确认 → 参考依据 → 视觉论点锁定 → 设计系统确认 → 规格文件生成 → HTML 构建 → 审计交付。

## 📜 设计律令（不可协商的硬规则）

```
字体家族       最多 2 个
颜色命名槽     恰好 5 个：primary / accent / bg / fg / muted
按钮高度       全页统一，最小 44px
卡片定义       shadow 或 border，二选一，不能混用
图标           SVG only，一个 family，尺寸只用 16/20/24px
正文行长       最多 70ch
正文最小字号   16px
间距基数       8px
对比度         正文 ≥ 4.5:1（WCAG AA）
```

完整律令见 `SKILL.md`。
